# DDR-001: GOAL.md Strict Validation at Startup

> Status: **Accepted**
> Date: 2026-03-18
> Related: ADR-ARCH-008 (start fresh), Domain Config module

## Context

GOAL.md is the central domain abstraction — it drives ingestion, prompt construction, generation targets, Coach evaluation, and output validation. A malformed GOAL.md can produce invalid training data that is expensive to detect after a 25-hour overnight run.

The architecture established a "fail fast" philosophy (ADR-ARCH-008: start fresh on re-run). The question is whether GOAL.md validation should be strict (halt on any issue) or lenient (warn and use defaults).

## Decision

Validate all 9 required GOAL.md sections at startup. Missing or malformed sections halt the pipeline immediately with a clear `GoalValidationError` identifying the section and the specific validation failure.

## Rationale

- A 25-hour overnight run with malformed config produces worthless output
- Fail-fast at startup costs seconds; fail-late costs hours
- Consistent with ADR-ARCH-008's philosophy of clean, complete runs
- No sensible defaults exist for domain-specific sections (system prompt, evaluation criteria)

## Alternatives Considered

- **Lenient with defaults:** Risk of silent generation of invalid training data. No sensible defaults for most sections.

## Consequences

- ML engineer gets immediate, actionable error messages when GOAL.md is incomplete
- No partial runs — either the config is valid or the pipeline doesn't start
- New domain onboarding requires all 9 sections before first run
