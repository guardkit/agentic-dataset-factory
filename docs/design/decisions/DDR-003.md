# DDR-003: 3-Turn Player-Coach Limit Before Discard

> Status: **Accepted**
> Date: 2026-03-18
> Related: ADR-ARCH-006 (sequential generation), ADR-ARCH-009 (non-deterministic), Generation module

## Context

The Player-Coach loop needs a maximum number of revision cycles per generation target. Too few turns may discard examples that could be improved; too many waste time on targets that will never pass.

With local model inference (~90 seconds per turn for a 120B model on GB10), the time cost per turn is significant over 1,000 targets.

## Decision

Maximum 3 turns per generation target: 1 initial attempt + 2 revision cycles. Configurable via `generation.max_turns` in `agent-config.yaml`, defaulting to 3.

## Rationale

- **Time budget:** 3 turns × ~90 seconds = ~4.5 minutes max per target. For 1,000 targets with ~5% rejection rate, adds ~4 hours. Acceptable for overnight runs.
- **Diminishing returns:** Empirical observation from similar systems suggests most improvements happen in the first revision. A third turn catches edge cases but further turns rarely succeed.
- **Configurable:** Can be increased for API mode (faster inference) or decreased for experimentation
- **Debugging value:** rejected.jsonl captures all 3 rejection reasons, enabling systematic analysis

## Alternatives Considered

- **5 turns:** More chances but diminishing returns. At ~90s/turn, 5 turns = 7.5 minutes max per target. Risk of spending too long on fundamentally flawed examples.
- **1 turn (no revision):** Fastest but loses the adversarial improvement benefit. Would increase rejection rate significantly.

## Consequences

- Examples that fail 3 turns are discarded to `rejected.jsonl` with full history
- Rejection rate should be monitored — consistently high rates suggest prompt improvement needed
- Can be tuned per domain or run via `agent-config.yaml`
