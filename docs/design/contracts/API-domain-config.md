# API Contract: Domain Config

> Module: `domains/`
> Generated by `/system-design` on 2026-03-18

## Responsibility

Owns the central domain abstraction. Each use case is a directory containing a GOAL.md and source documents. No code changes required to add a new domain.

## Directory Structure

```
domains/{domain-name}/
├── GOAL.md              ← 9 required sections (see schema below)
├── golden_set.jsonl     ← optional, for eval (downstream concern)
└── sources/
    ├── *.pdf            ← input documents processed by Docling
    └── ...
```

## GOAL.md Schema

All 9 sections are **required**. Startup validation fails fast on missing or malformed sections.

### Section 1: Goal

```markdown
## Goal

What behaviour the fine-tuned model should exhibit. Free-text description
of the target model personality, expertise, and interaction style.
```

**Validation:** Non-empty, minimum 50 characters.

### Section 2: Source Documents

```markdown
## Source Documents

| File Pattern       | Mode     | Notes            |
|--------------------|----------|------------------|
| mr-bruff-*.pdf     | standard | Digital PDFs     |
| aqa-mark-schemes/* | standard | Digital PDFs     |
| scanned-*.pdf      | vlm      | Scanned pages    |
```

**Validation:** Must contain a markdown table with columns: File Pattern, Mode. Mode must be `standard` or `vlm`. At least one row required.

### Section 3: System Prompt

```markdown
## System Prompt

The system prompt injected into every training example's messages[0].
Must be consistent across all examples in the dataset.
```

**Validation:** Non-empty, minimum 100 characters. Single block of text (no sub-sections).

### Section 4: Generation Targets

```markdown
## Generation Targets

| Category                          | Type      | Count | % of total |
|-----------------------------------|-----------|-------|------------|
| Literary analysis (single-turn)   | reasoning | 200   | 20%        |
| Essay feedback (multi-turn)       | reasoning | 250   | 25%        |
| Exam technique guidance           | reasoning | 150   | 15%        |
| Poetry comparative questions      | reasoning | 150   | 15%        |
| Factual recall / character / plot | direct    | 100   | 10%        |
| Terminology definitions           | direct    | 75    | 7.5%       |
| Encouragement / session mgmt     | direct    | 75    | 7.5%       |

Total: 1,000
Reasoning/direct split: 75% / 25%
```

**Validation:** Must contain a markdown table with columns: Category, Type, Count. Type must be `reasoning` or `direct`. Total count must be derivable. Reasoning percentage must be >= 70%.

### Section 5: Generation Guidelines

```markdown
## Generation Guidelines

Instructions for the Player agent — what to generate and how.
Includes style guidance, pedagogical approach, constraints.
```

**Validation:** Non-empty, minimum 100 characters.

### Section 6: Evaluation Criteria

```markdown
## Evaluation Criteria

The rubric the Coach uses. Each criterion has a name and description.

| Criterion          | Description                                              | Weight |
|--------------------|----------------------------------------------------------|--------|
| socratic_approach  | Tutor guides via questions rather than giving answers     | 25%    |
| ao_accuracy        | Correct application of assessment objectives             | 25%    |
| mark_scheme_aligned| Analysis aligns with AQA marking criteria                | 20%    |
| age_appropriate    | Language suitable for Year 10 student                    | 15%    |
| factual_accuracy   | No incorrect claims about texts or context               | 15%    |
```

**Validation:** Must contain a markdown table with columns: Criterion, Description. Criterion values must be valid Python identifiers (used as keys in Coach rejection schema `criteria_met`). At least 3 criteria required.

### Section 7: Output Schema

```markdown
## Output Schema

The exact JSON structure each training example must conform to.

```json
{
  "messages": [
    {"role": "system", "content": "<System Prompt>"},
    {"role": "user", "content": "<student message>"},
    {"role": "assistant", "content": "<tutor response>"}
  ],
  "metadata": { ... }
}
```
```

**Validation:** Must contain a JSON code block with a valid JSON object. Must have `messages` and `metadata` top-level keys.

### Section 8: Metadata Schema

```markdown
## Metadata Schema

| Field        | Type            | Required | Valid Values                                                    |
|--------------|-----------------|----------|-----------------------------------------------------------------|
| layer        | string          | yes      | behaviour, knowledge                                            |
| type         | string          | yes      | reasoning, direct                                               |
| ao           | array[string]   | yes      | AO1, AO2, AO3, AO4, AO5, AO6 (can be empty)                  |
| text         | string          | yes      | macbeth, a_christmas_carol, an_inspector_calls, ...             |
| topic        | string          | yes      | character_analysis, essay_feedback, exam_technique, ...         |
| grade_target | integer or null | yes      | 4, 5, 6, 7, 8, 9, null                                        |
| source       | string          | yes      | synthetic, aqa_derived, exam_board_adapted                     |
| turns        | integer         | yes      | Number of conversation turns (1+)                              |
```

**Validation:** Must contain a markdown table with columns: Field, Type, Required, Valid Values. All fields must have `Required = yes`.

### Section 9: Layer Routing

```markdown
## Layer Routing

| Layer       | Destination                         | Purpose                          |
|-------------|-------------------------------------|----------------------------------|
| behaviour   | output/train.jsonl                  | Teaches HOW the model responds   |
| knowledge   | output/rag_index/knowledge.jsonl    | Provides WHAT the model draws from|

Rules:
- behaviour: Examples demonstrating pedagogical technique (Socratic questioning, feedback patterns)
- knowledge: Examples primarily delivering factual content (quotes, themes, character analysis)
- If ambiguous, default to behaviour
```

**Validation:** Must contain a markdown table with `behaviour` and `knowledge` rows.

## Consumers

| Consumer | What it reads |
|----------|---------------|
| Entrypoint (`agent.py`) | Full `GoalConfig` for validation and orchestration |
| Prompt Engine (`prompts/`) | `generation_guidelines`, `evaluation_criteria`, `system_prompt` |
| Player Agent | `generation_guidelines` (injected into prompt) |
| Coach Agent | `evaluation_criteria` (injected into prompt, criterion names used in JSON response) |
| Ingestion Pipeline | `source_documents` (file patterns and Docling modes) |
| `write_output` Tool | `output_schema`, `metadata_schema`, `layer_routing` |
