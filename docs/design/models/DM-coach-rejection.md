# Data Model: Coach Rejection

> Generated by `/system-design` on 2026-03-18

## Entity: CoachVerdict

Structured JSON evaluation returned by the Coach agent for every training example submitted by the Player.

## Schema

```json
{
  "decision": "accept | revise",
  "score": 4,
  "layer_correct": true,
  "type_correct": true,
  "criteria_met": {
    "socratic_approach": true,
    "ao_accuracy": true,
    "mark_scheme_aligned": true,
    "age_appropriate": true,
    "factual_accuracy": true
  },
  "issues": [],
  "quality_assessment": "Strong example demonstrating effective Socratic pedagogy"
}
```

## Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `decision` | `"accept" \| "revise"` | Yes | Whether the example passes the quality gate |
| `score` | `int (1-5)` | Yes | Overall quality score. 1=poor, 3=adequate, 5=excellent |
| `layer_correct` | `bool` | Yes | Whether `metadata.layer` is correctly classified |
| `type_correct` | `bool` | Yes | Whether `metadata.type` matches `<think>` block presence |
| `criteria_met` | `dict[str, bool]` | Yes | Per-criterion pass/fail |
| `issues` | `array[Issue]` | Yes | Specific problems found (empty if accepted) |
| `quality_assessment` | `string` | Yes | Free-text overall assessment |

### criteria_met Keys

Keys are **dynamically derived** from GOAL.md `## Evaluation Criteria` section. The criterion `name` field becomes the key.

For the GCSE English tutor domain:
- `socratic_approach` — Guides via questions rather than giving answers
- `ao_accuracy` — Correct application of assessment objectives
- `mark_scheme_aligned` — Analysis aligns with AQA marking criteria
- `age_appropriate` — Language suitable for Year 10 student
- `factual_accuracy` — No incorrect claims about texts or context

### Issue Sub-Entity

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `criterion` | `string` | Yes | Which evaluation criterion this issue relates to |
| `severity` | `"blocking" \| "minor"` | Yes | Whether this issue alone would cause rejection |
| `description` | `string` | Yes | What is wrong |
| `suggestion` | `string` | Yes | How the Player should fix it |

## Acceptance Rule

An example is accepted when ALL of these conditions are met:

1. `decision == "accept"`
2. `score >= 3`
3. `layer_correct == true`
4. `type_correct == true`
5. No issues with `severity == "blocking"`

If any condition fails, the Player receives the full verdict for revision.

## Score Guide

| Score | Meaning | Typical Outcome |
|-------|---------|-----------------|
| 1 | Poor — fundamentally flawed | Revise (major rework needed) |
| 2 | Below standard — significant issues | Revise |
| 3 | Adequate — meets minimum bar | Accept (borderline) |
| 4 | Good — solid quality | Accept |
| 5 | Excellent — exemplary | Accept |

## Invariants

- `criteria_met` keys must exactly match GOAL.md Evaluation Criteria names
- All criteria must be present in `criteria_met` (no missing keys)
- If `decision == "accept"`, `issues` should be empty or contain only `minor` severity
- If `decision == "revise"`, at least one issue with `severity == "blocking"` should be present
- `layer_correct` and `type_correct` are independent structural checks, not part of criteria_met
