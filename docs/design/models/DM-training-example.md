# Data Model: Training Example

> Generated by `/system-design` on 2026-03-18

## Entity: TrainingExample

A single training example in ShareGPT format, produced by the Player agent and accepted by the Coach agent.

## Schema

```json
{
  "messages": [
    {
      "role": "system",
      "content": "<system prompt from GOAL.md>"
    },
    {
      "role": "user",
      "content": "<student question or response>"
    },
    {
      "role": "assistant",
      "content": "<think>\n<internal reasoning>\n</think>\n\n<visible response>"
    }
  ],
  "metadata": {
    "layer": "behaviour | knowledge",
    "type": "reasoning | direct",
    "ao": ["AO1", "AO2"],
    "text": "macbeth",
    "topic": "character_analysis",
    "grade_target": 7,
    "source": "synthetic",
    "turns": 1
  }
}
```

## Fields: messages

| Field | Type | Constraints |
|-------|------|-------------|
| `messages` | `array[Message]` | Non-empty. First element must be system role. |
| `messages[0].role` | `"system"` | Always `"system"` |
| `messages[0].content` | `string` | Must match GOAL.md System Prompt exactly |
| `messages[n].role` | `"user" \| "assistant"` | Alternating user/assistant pairs |
| `messages[n].content` | `string` | Non-empty |

### Think Block Rules

- If `metadata.type == "reasoning"`: last assistant message MUST contain `<think>...</think>` block
- If `metadata.type == "direct"`: last assistant message MUST NOT contain `<think>` block
- `<think>` blocks are internal reasoning â€” not shown to user at inference time
- Controlled by `enable_thinking` flag at serving time

### Multi-Turn Examples

Multi-turn examples have multiple user/assistant pairs after the system message:

```
system â†’ user â†’ assistant â†’ user â†’ assistant â†’ ...
```

All assistant turns in a reasoning example should include `<think>` blocks.
`metadata.turns` counts the number of user/assistant exchanges.

## Fields: metadata

| Field | Type | Required | Valid Values | Description |
|-------|------|----------|--------------|-------------|
| `layer` | `string` | Yes | `behaviour`, `knowledge` | Routes to output file |
| `type` | `string` | Yes | `reasoning`, `direct` | Whether `<think>` block is present |
| `ao` | `array[string]` | Yes | `AO1`-`AO6` (can be empty `[]`) | Assessment objectives covered |
| `text` | `string` | Yes | Domain-specific values from Metadata Schema | Source text/topic area |
| `topic` | `string` | Yes | Domain-specific values from Metadata Schema | Interaction type |
| `grade_target` | `int \| null` | Yes | `4`-`9` or `null` | Target grade level |
| `source` | `string` | Yes | `synthetic`, `aqa_derived`, `exam_board_adapted` | How the example was created |
| `turns` | `int` | Yes | `1+` | Number of conversation turns |

## Invariants

- `messages[0].role == "system"` always
- `metadata.type == "reasoning"` â†” assistant content contains `<think>` block
- `metadata.type == "direct"` â†” assistant content has no `<think>` block
- `metadata.turns == (len(messages) - 1) / 2` (excluding system message)
- `metadata.layer` determines output file routing
- All metadata field values must be in the valid values list from GOAL.md Metadata Schema

## Layer Classification

| Layer | Criteria | Destination |
|-------|----------|-------------|
| `behaviour` | Demonstrates pedagogical technique (Socratic questioning, AO-aligned feedback, grade calibration, guiding vs giving) | `output/train.jsonl` |
| `knowledge` | Primarily delivers factual content (quotes, themes, character analysis, mark scheme criteria) | `output/rag_index/knowledge.jsonl` |

If ambiguous, default to `behaviour`. The Player assigns the layer; the Coach validates via `layer_correct` field.
