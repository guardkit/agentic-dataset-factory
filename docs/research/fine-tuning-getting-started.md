# Fine-Tuning Getting Started Guide

**Gemma 4 31B on DGX Spark GB10 with Unsloth**

Date: April 2026
Status: Ready for first training run
Repo: `agentic-dataset-factory`

---

## Context

The GCSE English tutor dataset has been generated by the agentic dataset factory
(~2,500 examples, 94.8% acceptance rate, ShareGPT format). This guide covers
how to fine-tune Gemma 4 31B Dense on that dataset using Unsloth on the GB10.

### Why Gemma 4 31B Dense (not Nemotron 3 Nano)

| Factor | Gemma 4 31B Dense | Nemotron 3 Nano 30B-A3B |
|---|---|---|
| Architecture | Dense (all 31B active) | MoE (3B active per token) |
| Reasoning (AIME 2026) | 89.2% | Not published |
| Arena AI ranking | #3 open model | Not ranked |
| Unsloth support | Day-zero, bugs fixed Apr 8 | Supported |
| Fine-tuning suitability | Dense = simpler, higher quality ceiling | MoE = must avoid router layer, QLoRA risky |
| Inference speed | Slower (all params active) | Faster (3B active) |
| VRAM for QLoRA training | ~22GB | ~20GB |
| Licence | Apache 2.0 | NVIDIA Open Model Licence |

**Decision:** Use Gemma 4 31B Dense for fine-tuning. The quality gap is
substantial. If inference latency becomes a problem for real-time tutoring,
quantise the trained model (Q4_K_M GGUF) or evaluate the 26B MoE later.

### Training data compatibility

The existing `train.jsonl` has a 75% reasoning / 25% direct response split
with `<think>` blocks in assistant content. This transfers perfectly to Gemma 4:

- Gemma 4 has native thinking mode with the same `<think>` pattern
- Unsloth recommends ≥75% reasoning examples to preserve thinking ability
- The `gemma-4-thinking` chat template handles `<think>` blocks correctly
- ShareGPT format (`{messages: [{role, content}, ...]}`) is what Unsloth expects

No dataset changes are required.

## Step-by-step guide

### Step 1: Exit the current container

The playbook container was launched with `--rm` (auto-remove on exit) and
without volume mounts, so nothing persists. Exit it:

```bash
exit
```

The container is now gone. That's fine — we'll start a fresh one properly.

### Step 2: Prepare directories on the GB10 host

Before launching Docker, set up persistent directories on the host machine:

```bash
# Create working directories
mkdir -p ~/fine-tuning/data
mkdir -p ~/fine-tuning/output
mkdir -p ~/fine-tuning/scripts

# Copy training data to the data directory
# (adjust source path to wherever your train.jsonl lives on the GB10)
cp /path/to/agentic-dataset-factory/output/train.jsonl ~/fine-tuning/data/

# Copy the training script
# (transfer train_gemma4.py to the GB10 via scp or git)
cp /path/to/train_gemma4.py ~/fine-tuning/scripts/

# Verify
echo "Training examples:"
wc -l ~/fine-tuning/data/train.jsonl

echo "Script:"
ls -la ~/fine-tuning/scripts/train_gemma4.py
```

### Step 3: Launch Docker with volume mounts

This is the key difference from the playbook — we mount three directories so
data, model weights, and outputs survive container restarts:

```bash
docker run --gpus all \
  --ulimit memlock=-1 \
  --ulimit stack=67108864 \
  -it \
  -v ~/fine-tuning/data:/workspace/data \
  -v ~/fine-tuning/output:/workspace/output \
  -v ~/fine-tuning/scripts:/workspace/scripts \
  -v ~/.cache/huggingface:/root/.cache/huggingface \
  --entrypoint /usr/bin/bash \
  nvcr.io/nvidia/pytorch:25.11-py3
```

What each mount does:

| Mount | Purpose |
|---|---|
| `~/fine-tuning/data` → `/workspace/data` | Your `train.jsonl` |
| `~/fine-tuning/output` → `/workspace/output` | Checkpoints, merged model, GGUF |
| `~/fine-tuning/scripts` → `/workspace/scripts` | The training script |
| `~/.cache/huggingface` → `/root/.cache/huggingface` | Model weight cache (saves re-downloading ~60GB) |

**Note:** We deliberately omit `--rm` so the container persists if you detach.

### Step 4: Install dependencies (inside Docker)

Same as the playbook Step 4, run every time you create a new container:

```bash
pip install transformers peft hf_transfer "datasets==4.3.0" "trl==0.26.1"
pip install --no-deps unsloth unsloth_zoo bitsandbytes
```

This takes 2–3 minutes.

### Step 5: Verify the setup (quick sanity check)

Before committing to a full training run, do a 30-step test:

```bash
cd /workspace
python scripts/train_gemma4.py --max-steps 30
```

**What to watch for:**

- "Unsloth: Will patch your computer to enable 2x faster free finetuning" — confirms Unsloth loaded
- Model download progress (first run only, ~60GB for Gemma 4 31B)
- "Loaded N training examples from /workspace/data/train.jsonl" — confirms data loaded
- Sample formatted text printed — check it looks like proper Gemma 4 format
- "Verifying response-only masking..." — should show ~40–60% of tokens masked (the user/system parts)
- Training progress bar with loss decreasing
- Loss should be in the 1–3 range for the 31B model (not 100+)

**If loss explodes to 100+:** This means gradient accumulation is broken.
Update Unsloth (`pip install --upgrade --no-deps unsloth`) — the fix shipped
in v0.1.36-beta on April 8, 2026.

**If you get an IndexError on inference:** This is the `num_kv_shared_layers = 0`
bug. Unsloth has fixed this — make sure you have the latest version.

**If the process hangs (CPU 100%, GPU 5%):** This has been reported on Spark
with long runs. Start with `--max-steps 30` to confirm it works at small scale.
If it hangs on longer runs, try reducing `--max-seq-length 4096`.

### Step 6: Full training run

Once the sanity check passes, run the full training:

```bash
python scripts/train_gemma4.py \
  --epochs 1 \
  --save-steps 200 \
  --lr 2e-4
```

**Estimated timings on GB10 (128GB unified, QLoRA 4-bit):**

- ~2,500 examples, 1 epoch, max_seq_length=8192
- Expect roughly 3–6 hours depending on average sequence length
- Checkpoints saved every 200 steps to `/workspace/output/gcse-tutor-gemma4-31b/`

**Monitor memory:** In another terminal on the host:

```bash
watch -n 5 nvidia-smi
```

QLoRA 4-bit should use ~22–30GB of the 128GB unified memory. You have
substantial headroom.

### Step 7: Check the output

After training completes, you'll have three output directories:

```
/workspace/output/gcse-tutor-gemma4-31b/
├── lora-adapter/          # Small LoRA weights (~50-100MB)
│   ├── adapter_config.json
│   ├── adapter_model.safetensors
│   └── tokenizer files...
├── merged-16bit/          # Full merged model for vLLM (~60GB)
│   ├── model.safetensors
│   ├── config.json
│   └── tokenizer files...
└── gguf/                  # Quantised GGUF for llama.cpp/Ollama
    └── *.gguf
```

These are all on the host at `~/fine-tuning/output/` thanks to the volume mount.

### Step 8: Test the fine-tuned model

Quick inference test inside the same container:

```python
python -c "
from unsloth import FastModel
from transformers import TextStreamer

model, tokenizer = FastModel.from_pretrained(
    '/workspace/output/gcse-tutor-gemma4-31b/lora-adapter',
    max_seq_length=8192,
    load_in_4bit=True,
)

messages = [
    {'role': 'user', 'content': 'How does Shakespeare use dramatic irony in Romeo and Juliet Act 3 Scene 1? I am a Year 10 student studying for my GCSE English Literature exam.'}
]

inputs = tokenizer.apply_chat_template(
    messages,
    add_generation_prompt=True,
    enable_thinking=True,
    tokenize=True,
    return_dict=True,
    return_tensors='pt',
).to('cuda')

streamer = TextStreamer(tokenizer, skip_prompt=True)
model.generate(
    **inputs,
    max_new_tokens=512,
    use_cache=True,
    temperature=1.0,
    top_p=0.95,
    top_k=64,
    streamer=streamer,
)
"
```

**What to look for:**

- Does the model use `<think>` blocks before answering?
- Is the response pitched at GCSE level (not university level)?
- Does it reference AQA assessment objectives (AO1, AO2)?
- Is the tone encouraging and tutorial-like?

### Step 9: Serve via vLLM

To serve the merged model via vLLM (replacing or alongside your current
Qwen3-Coder-Next inference):

```bash
# Install vLLM (if not already in the container)
pip install vllm

# Serve the merged model
vllm serve /workspace/output/gcse-tutor-gemma4-31b/merged-16bit \
  --host 0.0.0.0 \
  --port 8000
```

Or for the GGUF via Ollama on the host:

```bash
# Create a Modelfile
cat > Modelfile << 'EOF'
FROM ~/fine-tuning/output/gcse-tutor-gemma4-31b/gguf/your-model.gguf
TEMPLATE """<|turn>user
{{ .Prompt }}<turn|>
<|turn>model
"""
PARAMETER temperature 1.0
PARAMETER top_p 0.95
PARAMETER top_k 64
EOF

ollama create gcse-tutor -f Modelfile
ollama run gcse-tutor
```

## Training parameters reference

| Parameter | Default | Notes |
|---|---|---|
| `--model-name` | `unsloth/gemma-4-31B-it` | The instruction-tuned variant |
| `--max-seq-length` | 8192 | Reduce to 4096 if OOM or hangs |
| `--no-4bit` | false | Use 16-bit LoRA (more VRAM, slightly better quality) |
| `--lora-r` | 8 | LoRA rank; 16 for higher quality, 4 for faster |
| `--lora-alpha` | 8 | Should equal lora-r at minimum |
| `--lr` | 2e-4 | Reduce to 2e-5 for multi-epoch runs |
| `--batch-size` | 1 | Increase if memory allows |
| `--grad-accum` | 4 | Effective batch = batch-size × grad-accum |
| `--epochs` | 1 | Start with 1; try 2-3 if loss plateaus |
| `--max-steps` | None | Overrides epochs; use 30-60 for quick tests |
| `--chat-template` | `gemma-4-thinking` | Use `gemma-4` if you want to suppress thinking |
| `--save-steps` | 100 | Checkpoint frequency |
| `--resume` | false | Resume from last checkpoint |
| `--skip-export` | false | Skip GGUF export step |

## Key decisions and trade-offs

### Why QLoRA 4-bit (not 16-bit LoRA or full fine-tune)?

The GB10 has 128GB unified memory — it could handle 16-bit LoRA for the 31B
model. However, QLoRA 4-bit is the right starting point because:

1. **First fine-tune ever** — get the pipeline working end-to-end first
2. **Faster iteration** — QLoRA trains faster, letting you experiment with
   hyperparameters before committing to a long 16-bit run
3. **Quality difference is minimal** — Unsloth's QLoRA implementation has
   negligible accuracy loss vs 16-bit for SFT tasks
4. **Easy to upgrade later** — once the pipeline works, just add `--no-4bit`

### Why `gemma-4-thinking` template?

Your dataset has 75% reasoning examples with `<think>` blocks. The
`gemma-4-thinking` template tells the model to expect and generate thinking
blocks. Using the non-thinking template would strip/ignore the reasoning
traces, defeating the purpose of the 75/25 split.

### Why `train_on_responses_only`?

This masks the loss on user/system tokens so the model only learns to generate
assistant responses. Without this, the model would also train on reproducing
user messages, which wastes capacity and can cause the model to echo prompts.

## Troubleshooting

### "CUDA out of memory"

1. Reduce `--max-seq-length` to 4096 or 2048
2. Ensure `--batch-size 1` (the default)
3. Confirm `use_gradient_checkpointing="unsloth"` is set (it is in the script)
4. Check nothing else is using the GPU (`nvidia-smi`)

### Loss is 100+ or NaN

Gradient accumulation bug — update Unsloth:
```bash
pip install --upgrade --no-deps unsloth unsloth_zoo
```

### Training hangs (CPU 100%, GPU idle)

Known Spark issue with long runs. Workarounds:
1. Reduce `--max-seq-length`
2. Add `--max-steps 500` and run multiple shorter sessions with `--resume`
3. Monitor with `watch nvidia-smi` to catch it early

### Model outputs garbage after export

Chat template mismatch at inference time. You MUST use the same template
(`gemma-4-thinking`) you trained with. The training script saves the tokenizer
config with the correct template in the output directory.

### "IndexError: list index out of range" during inference

The `num_kv_shared_layers = 0` bug. Fixed in Unsloth v0.1.36-beta. Update:
```bash
pip install --upgrade --no-deps unsloth unsloth_zoo
```

## What's next after the first successful fine-tune

1. **Evaluate** — Run a set of golden GCSE questions through the fine-tuned
   model and compare against the base model. Use Claude-as-judge if needed.

2. **Iterate on hyperparameters** — Try `--lora-r 16`, `--lr 2e-5`,
   `--epochs 2` based on eval results.

3. **Multi-subject expansion** — Once the pipeline is proven on English, add
   Maths, Science, French, Spanish, History domains. The training script works
   unchanged — just point `--data-path` at the new domain's `train.jsonl`.

4. **16-bit LoRA run** — If QLoRA quality is good but not great, try
   `--no-4bit` for a higher-quality run (will use ~60–70GB of the 128GB).

5. **Integrate with study-tutor inference** — Serve the fine-tuned model via
   vLLM alongside the existing RAG/ChromaDB layer for curriculum knowledge.
